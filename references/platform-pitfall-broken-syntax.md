# 小云雀平台兼容陷阱：破折号断裂句式

> **发现日期**：2026-06-15，实战60集审查  
> **严重度**：🔴 致命（小云雀NLP完全无法解析）  
> **影响范围**：任何使用「——」作为词汇分隔符的中文prompt  

## 问题描述

中文文本使用破折号`——`作为词汇分隔符，形成断裂句式：

```
原文：延炎——右手按在井沿——金焰——顺右臂——涌——入井
```

这种写法将主语、动作、宾语关系全部切断。小云雀的NLP分词器依赖连续的语义依存关系——当每个词被`——`隔离后，parser无法重建[主语→动作→宾语]的依存弧，导致prompt语义完全丢失。

## 检测正则

```regex
[\u4e00-\u9fff]+——[\u4e00-\u9fff]+——
```

匹配：连续中文词汇 + 破折号 + 连续中文词汇 + 破折号（两处以上即为断裂句式）

在终端批量检测：
```bash
grep -Pn '[\x{4e00}-\x{9fff}]+——[\x{4e00}-\x{9fff}]+——' *.md
```

## 破坏机制

| 阶段 | 发生了什么 |
|------|-----------|
| 分词 | 每个被`——`隔离的词被当作独立token，失去上下文 |
| 依存解析 | `延炎——右手——按——在——井沿` → parser找不到主语是谁、动词是什么 |
| 语义映射 | 小云雀内部将tokens映射到视觉元素时，找不到「谁」(延炎)+「做什么」(按)+「在哪」(井沿)的关联 |
| 生成结果 | 画面随机——角色可能错误、动作可能缺失、场景可能跳变 |

## 修复方法

### 标准修复（单句）

将断裂句式改为标准中文标点：

```
原文：延炎——右手按在井沿——金焰——顺右臂——涌——入井
修复：延炎的右手按在井沿，金焰顺着右臂涌入井中。
```

### 批量修复（sed）

```bash
# 简单替换：所有独立词汇间的「——」→「，」
sed -i '' 's/——/，/g' file.md

# 再手动合并逗号分隔的单字碎片
# 如：暗，金，光 → 暗金光
```

### AI修复（批量18集实测）

使用delegate_task分三路并行，每路处理6集。AI能理解语义，正确区分：
- 词汇分隔符用法的`——` → 替换为逗号/句号
- 真正解释说明用法的`——` → 保留

## 实战数据（《苍生》E26-E43修复）

| 批次 | 修复量 | 效果 |
|------|--------|------|
| E26-E31 (6集) | ~600处替换 | 可读标准中文，无残留断裂 |
| E32-E37 (6集) | 数千处→99%+清除 | 高质量散文式中文 |
| E38-E43 (6集) | 36,787处替换→逗号 | 叙述正文零残留 |

## 与SQ系统关联

- **SQ-破折号**：Phase 4新增审查关卡，在SQ0预检查阶段运行
- **SQ6**：prompt组装前必须通过此检测
- **SQ9**：提交前最后一道闸——任一断裂句式命中→回退SQ6修复

## 检测阈值

- 单集内`——`断裂句式≥5处 → WARN
- 单集内`——`断裂句式≥20处 → ERROR → 阻塞提交
- 全剧任意集有此问题 → 必须在提交前全量修复
