# 剧本创作工作流规范

> 技术规范·非创作方法论

---

## 一、禁止使用子程序撰写剧本正文

在本次会话（2026-06-27）中用户已明确纠正：**禁止使用delegate_task/subagent/子程序撰写剧本正文**。

### 原因

子程序生成的剧本存在以下系统性问题：
1. **前后逻辑不一致** — 子程序不知道全局剧情走向，A集的事件在B集被遗忘
2. **角色性格漂移** — 十七在第一集和第二集中回答的语气可能不一致
3. **对话线断裂** — 埋下的伏笔子程序不会自动回收
4. **数字/参数不统一** — 147米、七千四百二十道墨线等关键数据在不同子程序中可能不一致
5. **情感曲线扁** — 子程序无法感知全局的情感高低分布

### 正确的做法

1. **全剧本由主Agent在同一个上下文中逐集编写**。所有60集的内容在主Agent的同一轮执行中生成。
2. **先写大纲再填充细节**。先确保60集的大纲在逻辑上一致（主Agent完成），再逐集填充场景和对话。
3. **最后做全局一致性校验**。完成后检查关键参数（年份、数字、人物性格）在全剧中是否一致。

### 可使用子程序的场景（不涉及逻辑连贯性）

- 批量文件格式转换（MD→Word/PDF）
- 角色名批量替换（6+文件同时改名）
- 排版检查与修复
- 字数统计

## 二、角色名管理规范

当用户要求修改角色名时（如韩瑾→延炎、林小禾→林依依），必须同步更新**所有相关文件**：

1. 60集大纲（Word）
2. 参赛剧本（Word）
3. 参赛剧本优化版（MD）
4. 故事大纲（Word）
5. 10轮自检报告（MD）
6. 完整参赛稿（MD）

使用python-docx的run.text替换+MD全文替换。

## 三、评分修复迭代流程

从v1.0→v2.0→v3.0的迭代模式：

```
v1.0: 完成结构 → 10轮自检 → 发现16项缺陷
v2.0: 按P0→P1→P2→P3修复 → 大部分修复
v3.0: 五维深度升级 → 对薄弱维度逐一突破 → 目标75+
```

每次迭代完成后的验证方式：重新跑8维评分，对比前后得分。

## 四、60集→Word剧本批量生成

1. 用Python脚本一次性定义所有60集的场景/对白/尾钩数据
2. 使用write_ep()函数逐集写出
3. 使用python-docx或python-docx+MD中间格式转换为Word
4. 确保每集保留：场景标题、动作描述、角色对白、尾钩

避免做法：一集一集单独生成（断逻辑）。推荐做法：在一个Python数据结构中定义全部60集后一次性写出。
